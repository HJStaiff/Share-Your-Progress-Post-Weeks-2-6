# Progress post Weeks-2-6

### Overview
I'm currently working Worldbuilding Simulator. This project (made in the Godot engine) is set to be a map creation tool for dungeons and dragons games with a focus on planet scale simulations (like tectonics and climate). Our project lead, Zoe, is started this project to fill the gaps in ease of use where other map making tools are lacking. My role so far in this project has primarily been designing the UI elements and working with Zoe to figure out the way users would be interacting with the tool.

### Editor UI Layout
<img width="1860" height="1035" alt="image showing user interface layout" src="https://github.com/user-attachments/assets/3c5511a9-8425-4233-aeb1-128b496ccec0" />

The first task that I took on was creating the layout for the main scene where users will edit maps. I started by going through all the planned functions listed in our design document. Since this is supposed to have a similar workflow to existing mapmaking and image editing tools, I already sort of had an idea of what it would look like (something similar to photoshop, gimp, etc.). I knew users would have expectations for where buttons would be such as: tools on the left, options on top, main view in the middle. There are some planned features that don't exactly map to that layout that I had to fit in, like having four different main workflows (landmass, terrain, biomes, political) which ended up on the top bar. The second step I took with the ui was to update it depending on the current state. Depending on which mode you are in, the available tools on the left will change, and each tool has different parameters that will need to update on the right side. It took a while, but I came up with a system that should allow us to add brushes with their own parameters to the UI when we have them implemented. Currently, the layout and menu switching functionality is complete but the UI still needs a second pass to make it look more visually appealing. This shouldn't be too challenging since Godot lets you update the visuals of multiple nodes using a theme resource at once. 

### Radial menu
<img width="419" height="435" alt="image" src="https://github.com/user-attachments/assets/8e2e5208-4fc4-47c9-af12-aec6f008ef5c" />

The next task I worked on was creating the radial brush selection menu. This allows you to hold a button, move your mouse towards one option on the menu and release the button to select that option. This task was more concrete, I know how radial menus work, but Godot doesn't necessarily have something like this built in with the UI nodes so I had to be make it using draw functions and circle math. I knew for sure that someone had already made something like this so I looked online for help with the math. It's pretty straightforward, you need to keep track of the global position of the center of the circle and find the positions of everything else relative to that. There are already built in math functions in Godot for things like finding angles around a circle, and getting the positive modulus of that angle which made it easy to track the cursor around the menu. Then, once the radial menu button is released, it sends a signal with the selected option. Brushes aren't implemented in the main scene yet, however when they are they can be set to listen for this signal and update accordingly. Again, this radial menu needs a second pass for visuals, however I exposed the variables for color, size, radius to be modifiable in the editor. When we have a direction for Worldbuilding Simulator's visual style, the radial menu can be updated to match it.

### Challenges
The biggest challenge that I had while working on this so far wasn't even with the code but with git. I had been working on the radial menu on my own branch for a while and when I went to push the changes I ended up pushing them directly to the main branch. In the moment I wasn't really sure what had happened, I wasn't paying attention and I just went past what git was telling me since I assumed it would automatically handle things like normal. I attempted to revert the changes, but then I realized that I had gone one commit too far and reverted the last major PR. I brought it back and then cherry picked the radial menu commits into my branch and made a PR correctly. Everything was going fine until it happened again as I was trying to push my changes from the radius and intensity selection to my branch a week later. I realized that I had set my upstream branch incorrectly somehow and it was causing things to be pushed to main.  
Another challenge that I had working on this was when I was trying to review Zoe's PR adding the debug console and I was getting completely different behavior. Immediately it seemed like the problem was caused by something outside of Godot since I deleted and cloned the repo again, and there was no specific error message saying something was wrong. Our team uses linux, windows, and macOS all of which are supported by Godot. It took an hour or two of going through the code but we narrowed it down to the use of a colon character in the console log file name, which windows does not allow.

### Lessons learned
The main lesson I have learned from this, having used Godot before is that it really takes a significant amount of time to plan projects and get directions when you aren't the one with the ideas in your head. When I have done projects by my own I knew what I wanted to do and what it should end up looking like. With this project it has been essential to spend time talking about every detail, since many of my assumptions about how things should look or should be done are not 100% correct. The most helpful thing to have is a wireframe diagram, even a simple drawing in paint will do.
